## 🔖 重要说明：关于 M6/M7 的 Mock Data

**评估修正**: M6/M7 使用大量 Mock Data **不是问题**，是**正确的工程决策**。

**依据**: 全局开发策略 D010 (Frontend-First) + Phase 1 目标

**理由**:
1. **解耦开发**: 前端不被后端阻塞，可独立迭代
2. **早期验证**: UI/UX 在实现复杂后端前可验证
3. **API 契约锁定**: Mock Server 定义了清晰的接口规范
4. **符合阶段目标**: Phase 1 交付物就是"全 clickable 系统 (fake data, real UI)"

**结论**: M6/M7 的 18 项 mock 功能是**符合设计策略的预期状态**，不需要在 Phase 1 修复。Phase 2 后端完成后，按替换策略逐步迁移到真实 API。

## 📋 Mock Data 替换策略 (Phase 2 后期)

当 M1/M3/M4/M5 完成后，替换 mock data 时需遵循以下策略：

### 1. API 路由一致性
- **现状**: Mock Server 在 `/api/v1/*` 下提供假数据
- **目标**: M8 API Gateway 统一路由
- **风险**: 路由变更导致前端大规模重写
- **缓解**: contracts/ 已定义 OpenAPI Schema，确保 M8 遵循相同规范

### 2. 数据格式匹配
- **现状**: Mock Server 返回符合 contracts/ 的 Pydantic schema
- **目标**: 真实后端必须返回相同格式
- **验证**: 使用 `contracts/` 中的 schema 校验后端响应
- **测试**: 添加集成测试验证前后端数据格式一致性

### 3. 渐进迁移 (重要)
- **原则**: 不要一次性切换所有 API
- **策略**:
  1. **第一阶段**: 低风险功能
     - 切换设置页面加载 (Settings)
     - 切换语言切换功能
     - 验证数据格式、错误处理、加载状态
  2. **第二阶段**: 中等风险功能
     - 切换文档列表加载 (Documents/KG)
     - 切换用户列表加载
  3. **第三阶段**: 核心功能
     - 切换聊天功能 (Chat)
     - 切换文件上传+解析
     - 切换流式输出
- **回滚计划**: 每个阶段保留快速回滚到 Mock Server 的能力

### 4. Mock Server 保留价值
- **开发环境**: 继续用于前端开发、E2E 测试
- **演示环境**: 用于产品演示、客户试用
- **测试环境**: 用于隔离测试，不依赖不稳定后端
