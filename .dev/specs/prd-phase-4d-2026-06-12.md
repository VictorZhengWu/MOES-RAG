# PRD: Phase 4-D — 协作平台与知识沉淀

> **版本**: v1.0 | **日期**: 2026-06-12
> **依赖**: Phase 4-A/B/C 全部完成
> **定位**: 4-B/4-C 的深度延伸——案例库深度、模板管理、团队协作、项目间链接

---

## 1. 概述

### 1.1 背景

Phase 4-B 实现了项目归档 + 标记案例 + 基础检索。Phase 4-C 在 Deep Research 中加入了案例引用复选框。但存在四类明显缺口：

1. **案例库深度不足**：案例只能按标签检索，缺少结构化详情字段（关键挑战、解决方案、经验教训），没有相关性排序
2. **模板不可定制**：规范模板硬编码 5 个船型组合，用户无法新建、修改或基于项目复制模板
3. **无团队协作**：多人参与同一项目时只能通过外部渠道沟通，系统内无讨论串、@提及、通知
4. **项目间孤岛**：项目 B 无法引用项目 A 的结论，知识无法跨项目复用

Phase 4-D 解决这四类问题，使系统从"个人研究工具"升级为"团队合规协作平台"。

### 1.2 范围

| 功能 | 现状 | 4-D 目标 |
|------|------|---------|
| 案例库 | 标记 + 基础检索 | 结构化详情字段 + 相关性排序 + 案例引用追溯 |
| 模板管理 | 5 个硬编码模板 | 用户 CRUD 模板 + 基于项目复制 |
| 团队协作 | 无 | @提及 + 讨论串 + 通知（站内 + 邮件） |
| 项目间链接 | 无 | 结论引用 + 跨项目合规验证 + 知识复用 |
| 权限 | 仅 owner 可编辑 | 三级角色：Owner/Editor/Viewer |

---

## 2. 目标

- **G1**: 案例库增加结构化详情字段（关键挑战/解决方案/经验教训），支持相关性排序
- **G2**: 用户可创建、编辑、删除自定义规范模板；支持从现有项目复制为模板
- **G3**: 项目内支持 @提及团队成员、讨论串、站内通知
- **G4**: 项目间可引用结论——项目 B 的结论可标记"参考了项目 A 的结论 X"
- **G5**: 三级角色权限——Owner（完全控制）、Editor（可添加对话/文档）、Viewer（只读+评论）

## 3. 用户故事

### 3.1 案例库深度

#### US-001: 案例详情字段

**描述**: 标记为案例的项目可以填写结构化详情——关键挑战、解决方案、经验教训。

**验收标准**:
- [ ] `PATCH /api/v1/projects/{id}` 新增字段: `case_challenge`, `case_solution`, `case_lessons`
- [ ] 案例项目详情页增加 "Case Study Details" 区域（三段式展示）
- [ ] 字段均为可选，未填写时显示 "—"
- [ ] 案例详情可通过 `GET /cases` API 返回

#### US-002: 案例相关性排序

**描述**: 案例检索时按相关性排序——同船型优先、同入级社优先、关键词匹配。

**验收标准**:
- [ ] `GET /api/v1/projects/cases?query={keyword}&vessel_type={type}` 支持过滤参数
- [ ] 排序算法: 同船型 +3 分，同入级社 +2 分，标题关键词匹配 +1 分，按总分降序
- [ ] 无匹配时返回全部案例（不排序优先）

#### US-003: Deep Research 案例引用追溯

**描述**: Deep Research 在报告中明确标注引用了哪个案例的哪个结论。

**验收标准**:
- [ ] 研究计划中的案例子问题结果追加到 `agent_web` 输出
- [ ] 报告 §6 引用追溯中，案例引用格式为: `[Case-N] Project "XXX" — conclusion: "YYY"`
- [ ] 案例引用可点击跳转到案例详情页

### 3.2 模板管理

#### US-004: 自定义规范模板 CRUD

**描述**: 用户可以创建、编辑、删除自己的规范模板。

**验收标准**:
- [ ] `POST /api/v1/templates` — 创建模板（name + vessel_type + primary_class + regulation_list）
- [ ] `GET /api/v1/templates` — 列出用户模板 + 内置模板
- [ ] `PATCH /api/v1/templates/{id}` — 编辑模板
- [ ] `DELETE /api/v1/templates/{id}` — 删除（仅自定义模板，内置模板不可删）
- [ ] M7 Admin 可管理全局内置模板

#### US-005: 从项目复制为模板

**描述**: 完成的项目可一键保存为规范模板，供后续项目复用。

**验收标准**:
- [ ] `POST /api/v1/projects/{id}/save-as-template` — 提取项目的 regulation_list 创建模板
- [ ] 模板名称默认为项目名称 + " (Template)"
- [ ] 项目创建对话框的模板下拉同时显示内置 + 自定义模板

### 3.3 团队协作

#### US-006: @提及团队成员

**描述**: 在结论、问题、评论中可以 @提及团队成员，对方收到通知。

**验收标准**:
- [ ] 输入 `@` 弹出团队成员下拉列表（从 `team_members` 字段读取）
- [ ] @提及格式: `@username`，存储为 `<mention>@username</mention>`
- [ ] 被提及者收到站内通知 + 可选择邮件通知

#### US-007: 讨论串（Thread）

**描述**: 每个结论和问题可以有独立讨论串。

**验收标准**:
- [ ] `POST /api/v1/projects/{id}/conclusions/{cid}/comments` — 添加评论
- [ ] `GET /api/v1/projects/{id}/conclusions/{cid}/comments` — 查看讨论串
- [ ] 评论支持 @提及
- [ ] 新评论通知相关成员

#### US-008: 通知中心

**描述**: 顶部导航栏显示未读通知数，点击展开通知列表。

**验收标准**:
- [ ] `GET /api/v1/notifications` — 列出当前用户的通知（分页）
- [ ] `PATCH /api/v1/notifications/{id}/read` — 标记已读
- [ ] 通知类型: `mention`（被@提及）、`comment`（新评论）、`assignment`（被分配问题）
- [ ] 通知图标显示未读数 badge
- [ ] 点击通知跳转到相关页面

### 3.4 项目间链接

#### US-009: 结论跨项目引用

**描述**: 项目 B 的结论可以标记"参考了项目 A 的结论 X"。

**验收标准**:
- [ ] 创建结论时可通过 `linked_projects` 字段引用其他项目的结论
- [ ] `GET /api/v1/projects/{id}/conclusions` 返回时包含 `referenced_by`（哪些项目引用了这个结论）
- [ ] 结论详情显示: "Referenced by: Project A, Project C"
- [ ] 引用关系存储为双向（被引用方可见引用方）

#### US-010: 跨项目合规验证

**描述**: 如果项目 B 引用了项目 A 的已验证合规条款，可标记为"已验证（通过项目 A）"。

**验收标准**:
- [ ] 合规条款新增状态: `verified_by_reference`（通过引用其他项目的已验证结论）
- [ ] 点击引用来源可跳转到源项目的合规条款
- [ ] 引用链可视化: A.§6.2 ← verified by "Project A" ← referenced by "Project B"

### 3.5 权限管理

#### US-011: 三级角色系统

**描述**: 项目支持 Owner、Editor、Viewer 三级权限。

**验收标准**:
- [ ] Owner: 完全控制（CRUD 项目、管理成员、删除）
- [ ] Editor: 可添加对话、文档、问题、结论、评论
- [ ] Viewer: 只读 + 可评论
- [ ] `PATCH /api/v1/projects/{id}/members` — Owner 管理成员和角色
- [ ] M7 Admin 可查看所有项目（不受权限限制）
- [ ] API 层面强制权限检查（中间件）

### 4.4 权限中间件

```python
# m8-api-gateway check_project_access()
def check_project_access(project_id, user_id, required_role='viewer'):
    project = await pm.get_project(project_id)
    if project['owner_id'] == user_id:
        return True  # Owner has full access
    member = find_member(project['team_members'], user_id)
    if not member:
        raise HTTPException(403)
    role_hierarchy = {'viewer': 1, 'editor': 2, 'owner': 3}
    if role_hierarchy[member['role']] < role_hierarchy[required_role]:
        raise HTTPException(403)
    return True
```

## 5. 非目标（4-D 不做）

- ❌ 实时协作（WebSocket 多人同时编辑）— 技术复杂度高，延后
- ❌ 邮件通知发送（SMTP 配置已就绪，但邮件模板和发送调度延后）
- ❌ 项目间合规引用链可视化（D3.js 图）— 延后到独立迭代
- ❌ 团队看板（多项目聚合视图）— 延后
- ❌ 全文搜索通知内容 — 延后
